Проект реализует регистрацию пользователей, вход, выход, редактирование профиля, мягкое удаление аккаунта и разграничение прав доступа через собственные роли и правила.

## Возможности
- регистрация пользователя;
- вход по email и паролю;
- выход из аккаунта;
- редактирование профиля;
- мягкое удаление аккаунта;
- собственная модель пользователя;
- собственные роли и правила доступа;
- mock-страницы бизнес-ресурсов;
- страница управления правами;
- API для получения и изменения правил доступа;
- обработка ошибок `401` и `403`.

## Что не используется
В проекте не используется стандартная система авторизации Django:
- `django.contrib.auth`;
- стандартная модель `User`;
- `Group`;
- `Permission`;
- `AuthenticationMiddleware`;
- Django admin;
- DRF.

Используются только базовые возможности Django: ORM, views, forms, templates, sessions и migrations.
## Основные модели
### User
Собственная модель пользователя.
Хранит email, ФИО, хеш пароля, статус активности и дату мягкого удаления.
При удалении аккаунта пользователь не удаляется из базы, а получает: is_active = False
После этого он больше не может войти в систему.

### Role
Роль пользователя.
Примеры:
admin
manager
reader

### UserRole
Связь пользователя и роли.

### Resource
Ресурс, к которому проверяется доступ.
В проекте используются:
documents
reports
payments
access-rules


### Action
Действие над ресурсом.
В проекте используются:
read
manage

### PermissionRule
Правило доступа.
Формат правила:
role + resource + action
Если активное правило существует, доступ разрешен.  
Если правила нет, доступ запрещен.

## Логика доступа
1. Пользователь открывает защищенный ресурс.
2. Middleware пытается найти пользователя в session.
3. Если пользователя нет — 401 Unauthorized.
4. Если пользователь есть, проверяются его роли.
5. Если активное правило доступа найдено — ресурс открывается.
6. Если правила нет — 403 Forbidden.

## Установка и запуск
Установить зависимости:
pip install -r requirements.txt

Применить миграции:
python manage.py migrate

Создать тестовые данные:
python manage.py seed_demo

Запустить сервер:
python manage.py runserver

Открыть приложение:
http://127.0.0.1:8000/

## Тестовые пользователи

После выполнения команды `seed_demo` будут созданы пользователи:

Логин, пароль, роль
 `admin@example.com` - `Admin123!` - admin 
 `manager@example.com` - `Manager123!` - manager 
 `reader@example.com` - `Reader123!` - reader 
 `inactive@example.com` - `Inactive123!` - неактивный 

## Права пользователей
### admin
Имеет доступ к документам, отчетам, платежам и управлению правилами доступа.

### manager
Имеет доступ к документам и отчетам.

### reader
Имеет доступ только к документам.

### inactive
Не может войти в систему, потому что его аккаунт неактивен.


## API

API дступно только пользователям с правом:
access-rules + manage


## Кратко о реализации
После входа пользователя его ID сохраняется в session:
request.session["custom_user_id"] = user.id

Middleware получает пользователя из session и записывает его в:
request.current_user

Дальше доступ проверяется через таблицу `PermissionRule`.

Если пользователь не найден — возвращается `401`.  
Если пользователь найден, но права нет — возвращается `403`.  
Если право есть — ресурс открывается.

## Для защиты

В проекте реализована собственная система аутентификации и авторизации без использования `django.contrib.auth`.

Пользователи хранятся в собственной модели `User`.  
Авторизация построена на моделях:

Role
UserRole
Resource
Action
PermissionRule

Доступ проверяется по правилу:
роль + ресурс + действие

Для демонстрации работы системы реализованы mock-страницы документов, отчетов и платежей, а также страница управления правилами доступа.
